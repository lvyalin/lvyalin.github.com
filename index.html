高效的团队开发环境

发表于 2015-06-01   |  

团队的开发环境是能够体现团队工作效率的一个重要组成部分。一个好的高效的开发环境能提升效率，减少开发过程中低级失误。

团队开发环境遇到的问题

1.无法快速搭建开发环境。尤其是项目比较庞大的情况下。体现在新同事入职，启动新项目。
2.环境的隔离差。许多人共用一个服务比如mysql，nginx等。一个同学出现低级失误，比如修改nginx错误，drop mysql库，大家都歇菜了。还有测试和开发环境的隔离。
3.线下环境和线上环境不一致。比如mysql版本，php版本，php扩展的安装等等。这个原因是线上服务一直有运维人员维护升级而开发环境没有
4.测试工程师希望能根据svn地址和版本号快速构建出一个纯洁的测试环境(不被开发染指)。且每次测试完成修复bug后，能迅速销毁重新构建，这就意味着要简单。

理想的团队开发环境

每个人有自己的多套环境（每个人处于多个分支开发中），且环境配置和线上能保持一致。能够快速简易搭建环境。

我们的解决方案

vagrant

Create and configure lightweight, reproducible, and portable development environments.
是一款用来构建虚拟开发环境的工具，非常适合 php/python/ruby/java 这类语言开发web应用，“代码在我机子上运行没有问题”这种说辞将成为历史。

如何使用Vagrant构建开发环境

构建基础box，将开发所需要的服务，比如php，mysql，redis等安装到一个机器上，打包成box。做成团队开发环境。
需要搭建新环境只需要执行三步就行
1.vagrant box add web
2.vagrant init web
3.vagrant up
一个新的环境就有了，基础box可以由运维维护。按时升级。
我们团队服务端开发，全部迁移了到vagrant，为了方便开发，弄了两台内网服务器，做了vagrant，当开发有环境需求时，直接构建并且分配虚拟机(写个shell脚本)。当一个人有在多个分支下开发的时候（小公司很常见），直接分多台虚拟机给他。这块将来可以做一个内网web服务。开发人员在一个页面上申请PHP开发环境释放PHP开发环境就OK了。离这一步其实就差个页面了。

测试的需求，其实环境有了，差的就是代码和数据了，我们本身应用框架有脚手架功能，能够一键部署代码和导入数据。所以就直接引入了。测试只需要知道svn地址和版本号，还有开发上送的数据库等数据文件就能直接构建环境测试了。测试终于不用担心在测试的过程中，研发偷偷改代码了。

代码发布

发表于 2015-05-30   |  

为什么要谈代码发布

互联网服务一般来说都是要求高可用运行，可用性能达到99.99%以上，大型公司要求是5个9甚至更多，但是往往业务需要不断快速迭代以在市场竞争中成功。我待过公司有时每天发布十几次之多，尤其在快速发展的核心业务线上更是如此。与之相对的是每次上次都有可能引发故障，而且往往90%以上的线上故障都是由变更引发的。我们还需要用户体验上平滑的无感知的发布。

平滑发布

我们可以假设一个常见的架构，我们有1000台web服务器，请求流量通过HA分发到所有服务器上，同时有CDN节点部署静态资源。
我们发布的过程中可能遇到的几个问题：
1、部署的过程中产生了先后顺序，用户请求的时候有的是完成部署的服务器有的是没有部署的服务器。
2、静态资源的问题，因为存在浏览器缓存的问题，导致用户访问出错。
3、CDN更新问题。也会导致用户访问静态资源出错。

关于前端资源缓存的问题，基本的解决思路是非覆盖式发布。用文件的摘要信息来对资源文件进行重命名，把摘要信息放到资源文件发布路径中，这样，内容有修改的资源就变成了一个新的文件发布到线上，不会覆盖已有的资源文件。上线过程中，先全量部署静态资源，再灰度部署页面，整个问题就比较完美的解决了。可以参考百度实现方案http://fis.baidu.com/index.html

关于代码部署顺序问题。基本的解决思路是把代码推送和代码变更分离。将代码打包推送到服务器上。当所有机器都推送完成后，统一进行变更。具体做法是将打包代码时文件夹命名为项目名+svn版本号。当所有机器推送完成，并且包md5校验完成后，统一发送变更信号，系统接受信号后，将推送的代码文件夹建立软连到正式文件夹。完成变更。如果要求更加精准可以将发布定时变更信号（即在12:00:00统一变更）。前者误差在100ms内1000台机器可以完成变更，后者误差取决于机器时间的校对，如果时间校对正确可以保证基本没有误差。
其他的实现方式
第一种，让同一个用户的多次请求永远都在同一台机器上就不会有部署先后问题。即种会话保持cookie。但是问题往往不是这么简单。如果是接口请求怎么办？比如你维护了一个接口调用，这种请求是服务端调用根本不是用户的请求。还是存在问题。
第二种，通过调整负载均衡设备将部分设备下线，下线后部署。部署完成后，将下线机器上线，将线上机器下线部署。直到全部部署完成。这样问题是会导致发布过程中服务容量缩水。上线过程漫长。

减少或者尽量避免发布导致的故障

互联网公司一般会采用技术上和必要流程来控制上线故障。不能极端的相信只靠流程能解决一切问题。鄙视下线下走纸质审批上线的公司。有种回到石器时代的感觉。

火车模型

火车模型是指把上线过程看做一次火车旅行。旅行中有许多站点（检查项），必须停靠这些站点（完成检查）才能向下一个进发，直到终点（成功）。也可以对应出简化版，以满足重大变更上线，和修改一个简单bug这两种情况。

重大变更上线

1.上线准备阶段
开发核查checklist，完成后向项目负责人发送上线申请。
2.上线实施阶段
QA确认测试通过。运维确认资源到位。产品确认上线时间点。代码自查及review通过，单元测试通过。
全部确认后，发布到准生产环境。
在准生产环境QA确认测试通过。如有准生产环境发送的性能邮件。排查完成确认。如有准生产环境报警。排查完成确认。
全部确认后，正式上线。
3.上线验证阶段及回退
通过观察日志错误，测试人员线上跟踪，进行验证。如果出问题按照预定回退方案回退，一般回退方案为代码回退到上一版本。
4.监控及其余
新功能和新项目添加监控。上线后要确认添加相关监控并能接收到相关报警。

简单变更上线

1.上线准备阶段
开发向项目负责人发送上线申请。
2.上线实施阶段
QA或者开发确认测试通过。代码自查通过，有单元测试的话单元测试通过。
全部确认后，发布到准生产环境。
在准生产环境QA或开发确认测试通过。如有准生产环境发送的性能邮件。排查完成确认。如有准生产环境报警。排查完成确认。
3.上线验证阶段及回退
通过观察日志错误，线上跟踪，进行验证。如果出问题按照预定回退方案回退，一般回退方案为代码回退到上一版本。
4.监控及其余

自动化测试

当产品快速迭代，上线频繁的时候，每次上线如果只测试修改的功能点，可能导致覆盖面不够，上线后出现bug。但是如果做所有功能测试即使是全部主功能测试，时间和人力成本也是产品不能接受的。对于服务端来说，覆盖所有功能或者主功能的自动化测试至关重要，工程师只要花几分钟就可以将所有流程测试一遍。以保证代码逻辑的正确性。对于前端来说可以使用selenium之类浏览器驱动的自动化测试工具完成web功能测试和浏览器兼容测试。

准生产测试

之所以要做准生产测试，是因为线上环境和线下环境区别导致的。环境差异可能是，数据库、接口、缓存、第三方服务比如短信和银行接口。比如数据库结构不一致，接口变化，配置错误，扩展缺失。这些都可以导致应用故障。也就是非线上环境无法100%模拟线上环境，这就是准生产测试的意义用于排除环境差异上的问题。准生产测试服务器和正式服务器区别在于没有在负载均衡设备上，用户不能访问，其他完全一致。如果在准生产环境下测试通过，那么在生产环境上一定是成功的。

回退方案

即使我们做了再多的工作也不可能避免发布故障发生。回退方案就是在故障真实发生的情况下，最大程度的减少危害的方式。所以回退方案一定要制定，并且有自动化实施脚本。发生变更导致的故障时，第一时间对某个时间段内的上线做回退。不要试图线上解决问题。在回退结束后去解决问题，再走上线流程。自动化实施回退的意义在于不要认为发生故障时，程序员手忙脚乱的能回退成功，况且在不经常回退或者第一次回退的情况下。

支付宝携程所想

发表于 2015-05-28   |  

支付宝携程服务接连出现问题，本来不打算多说些什么来着，其实大家都挺明白都有苦衷。但是这次思考下，如果我们遇到类似问题怎么办？

支付宝的事

说实在的真的挺难处理。现在互联网服务说是分布式集群服务，但是或多或少还是存在难以分布的单点的。支付宝全国都有机房，估计北京的某个机房光纤挂断了，大家连知道都不知道。一旦某重要难以分布式部署的服务出现问题，就是服务可能短时间中断。

现在大部分中小型互联网企业实际上很难做到基本真正意义上的同城双机房双活。（比如双机房数据存储的同步，机房之间专线的高可用，数据CAP），但是大家基本可以做到web服务器多机房多点无状态部署，服务分级管理及服务隔离等，数据备份和失效转移，缓存就近部署等。

携程的事

据传是某技术人员由于人为操作失误不知道删了一个什么根目录文件于是乎导致灵异事件发生,线上数据全部被删，再次发布依旧被删。

对于高可用的互联网服务来说，携程的事的确挺悲哀的。
互联网企业最起码应该能保证，快速部署最基本服务单元的能力。其实就是临时搞来一批裸机，摘掉某些数据备机，能在30分钟内，快速搭建一套降级后基本核心服务。然后将流量切到该服务单元。这种能力在平时就是服务伸缩性的一种体现。搞活动流量激增，分分钟将瓶颈处的资源快速水平扩容也类似，不过这种扩容更简单。

对于中小互联网企业。假如出现不明原因的宕机或者服务不可用。排查问题的同时，第一时间一天时间内所有代码和变更回滚（此处应做到自动化）。为啥要回滚，因为99%严重的故障都是自己人作死，或者新上线代码问题或者有配置变更诸如此类。程序员每次上线都紧张不是没有道理的。如果还是有问题，如果有明显的资源使用紧张还可以逐步服务降级，服务降级就是去除非核心服务比如论坛。（此处要有事前规划），为啥要服务降级，一般来说大家都有坑死队友的潜力。没有资源紧张的征兆就快速拿资源构建降级后基本核心服务，验证之后切流量之类。代码和变更回滚可以在5分钟内解决到99.9%的故障。后者可以在30分钟内使核心服务恢复正常。

吐个槽，别人的灾难不是你快乐的盛宴

最后说句驱动公司技术发展的主要力量还是公司的业务发展。如果你公司PV都不过亿就跟着支付宝搞什么两地三中心了。一个机房做好冗余也不错。毕竟挖断光纤这事不是年年都能遇上的。
